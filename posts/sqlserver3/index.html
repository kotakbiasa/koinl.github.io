Rangkaian pengoptimalan kinerja pertama

Misalkan sebuah perusahaan transportasi nasional telah membuat sistem OLTP skala besar dan membuat gudang data di atas sistem tersebut. Ada tabel data berikut dalam sistem OLTP dan gudang data:
Detail transportasi (ID pesanan transportasi, ID stasiun pengirim, ID tujuan akhir, ID kargo, berat kargo, harga transportasi, tanggal pengiriman)
Tabel ringkasan 1 (ID stasiun pengirim, ID tujuan akhir, ID kargo, tanggal pengiriman, berat total, total harga angkutan)
Tabel ringkasan 2 (ID stasiun pengirim, ID area tujuan, ID kargo, tanggal pengiriman, berat total, total harga angkutan)
Tabel Ringkasan 3 (ID Stasiun Pengirim, ID Stasiun Tujuan Akhir, ID Kargo, Bulan Pengiriman, Berat Total, Total Harga Pengangkutan)
Tabel ringkasan 4 (ID area pengiriman, ID area tujuan, ID kategori kargo, tanggal pengiriman, berat total, total harga angkut)
Perusahaan mengelola sekitar 100 stasiun pengangkutan dengan sekitar 500 jenis barang dalam 10 kategori. Setiap tabel ringkasan memiliki kode master, dan setiap tabel memiliki strategi pemeliharaan yang masuk akal, sehingga data tetap konsisten setelah setiap pemeliharaan. Ada tampilan V yang memiliki frekuensi akses tinggi. Pola hasil querynya adalah (ID wilayah pengiriman, ID tujuan akhir, bulan pengiriman, total berat, total harga angkut). Tampilan ini sekarang menggunakan tabel ringkasan 1 sebagai sumber data penghitungan. Setelah dilakukan pemantauan, ditemukan bahwa tabel ringkasan 1 terlalu sering diakses sehingga menyebabkan kinerja sistem secara keseluruhan menurun, sedangkan tabel ringkasan lainnya lebih jarang diakses. Tanpa menambahkan tabel ringkasan dan indeks, berikan rencana pengoptimalan untuk meningkatkan kinerja layanan sistem dan jelaskan alasannya secara singkat.

[Ide pemecahan masalah]
　　Ada dua jenis pemrosesan data dalam sistem komputer: pemrosesan operasional dan pemrosesan analitis, juga dikenal sebagai OLTP (pemrosesan transaksi online) dan OLAP (pemrosesan analitis online). Pemrosesan operasional, juga disebut pemrosesan transaksi, mengacu pada operasi harian pada koneksi database, biasanya menanyakan atau memodifikasi satu atau sekelompok catatan, seperti sistem tiket kereta api, sistem penyetoran dan penarikan bank, sistem manajemen pengumpulan pajak, dll. Sistem ini memerlukan respons cepat terhadap permintaan pengguna dan memiliki persyaratan tinggi pada integritas data, integritas, dan throughput transaksi. Dikombinasikan dengan permasalahan dalam pertanyaan ini, tampilan itu sendiri memiliki jumlah kunjungan yang tinggi, dan hanya tabel ringkasan 1 yang digunakan sebagai sumber data penghitungan, sedangkan tingkat akses tabel ringkasan lainnya rendah, sehingga mengakibatkan pemanfaatan sumber daya yang tidak masuk akal. Oleh karena itu, pertanyaan ini mengkaji masalah penjadwalan sumber daya dalam pemrosesan transaksi online.
[jawaban referensi]
　　Karena frekuensi akses skema tabel ringkasan 1 dan tampilan sangat tinggi, dan sumber data tampilan berasal dari tabel ringkasan 1, dan karena tingkat akses tabel ringkasan lainnya rendah, Anda hanya perlu mengikat sumber data tampilan ke tabel ringkasan 3, karena tabel ringkasan 3 juga dapat memenuhi mode keluaran tampilan. Hal ini tidak hanya meningkatkan kecepatan akses data pada tabel ringkasan 3, namun juga mengurangi kecepatan akses data pada tabel ringkasan 1. Kinerja sistem dan kinerja layanan telah sangat dioptimalkan. Dan karena total ada sekitar 500 jenis barang dan 10 kategori, Anda dapat membuat tampilan lain untuk mengikat sumber data ke tabel ringkasan 4, sehingga Anda dapat memanfaatkan sepenuhnya informasi data di tabel ringkasan 4 dan lebih mengoptimalkan kinerja sistem.

Rangkaian optimasi kinerja pemulihan keenam

Karena perkembangan bisnis, sistem asli mal online tidak dapat memenuhi kebutuhannya dengan baik, sehingga mengadopsi sistem manajemen bisnis baru. Sistem ini menggunakan sistem manajemen basis data SQL Server 2008. Sebelum sistem ini online, data asli mal perlu diimpor ke sistem baru. Sistem asli menggunakan SQL Server 2000, dan struktur datanya tidak sepenuhnya konsisten dengan sistem baru.
Oleh karena itu, data perlu diimpor dari SQL Server 2000 ke SQL Server 2008. Untuk memastikan konsistensi data, bisnis harus ditangguhkan selama proses impor data dan harus diselesaikan dalam waktu 3 jam.
(1) Selama proses impor data asli ke sistem baru, pelaksana menemukan bahwa volume data asli sangat besar dan memerlukan waktu empat jam untuk mengimpor data, yang tidak dapat diterima oleh bisnis. Setelah analisis, seorang insinyur percaya bahwa I/O database selama proses impor data sangat tinggi, namun program untuk mengimpor data itu sendiri memiliki tingkat hunian sumber daya sistem yang sangat rendah. Insinyur menyarankan untuk mengubah mode pemulihan data selama proses impor data dari mode "Selesai" ke mode "Sederhana" untuk meningkatkan kecepatan impor data; sementara teknisi lain percaya bahwa metode ini mungkin tidak meningkatkan kecepatan impor data dan juga dapat menyebabkan hilangnya data, jadi metode ini tidak disarankan.
Harap analisis apakah metode ini dapat meningkatkan kecepatan impor data dan berikan alasannya, serta analisis risiko kehilangan data akibat operasi ini.
(2) Setelah berhasil mengimpor data historis, sistem online dengan lancar. Pada minggu pertama operasi online, ditemukan bahwa penggunaan CPU server database sangat tinggi, mencapai hampir 90%, mencapai 100% selama periode puncak, dan penggunaan memori sistem mencapai 90%, tetapi I/O sistem sangat ringan. Staf bisnis melaporkan bahwa sistem beroperasi sangat lambat. Untuk meningkatkan kecepatan berjalan sistem.
Tanpa memodifikasi aplikasi, kedua insinyur tersebut memberikan solusi berbeda:
Ⅰ Tambahkan 2 CPU ke server untuk mengatasi masalah penggunaan CPU yang tinggi;
Ⅱ Gandakan memori server untuk mengatasi masalah penggunaan memori yang tinggi.
Mengingat biayanya, hanya satu rencana yang dapat dilaksanakan pada tahap ini. Harap tunjukkan pilihan mana yang lebih masuk akal dalam situasi saat ini dan berikan alasannya.

(1) Metode ini dapat meningkatkan kecepatan impor data.
Alasan: I/O sistem ini sangat tinggi. Setelah memodifikasi model pemulihan, sistem meminimalkan overhead log dan meningkatkan kecepatan impor. Karena model pemulihan hanya diubah selama proses impor data, tidak ada risiko kehilangan data.
(2) Pilihan pertama lebih masuk akal. Alasan: SQL Server 2008 menggunakan metode buffering data di memori, jadi wajar jika penggunaan memori relatif tinggi, dan tidak ada masalah dengan I/O pada tahap ini, yang menunjukkan bahwa memori memenuhi permintaan. Tingkat penggunaan CPU pada tahap ini sangat tinggi, menunjukkan sumber daya komputasi CPU tidak mencukupi, sehingga menambah jumlah CPU efektif dalam menyelesaikan masalah.

Rangkaian optimalisasi kinerja kedelapan

Misalkan sistem manajemen supermarket menggunakan sistem manajemen database SQL Server 2008. Server database ini memiliki 2 CPU, memori 16GB, dan disk 2TB. Satu tahun setelah online, pengguna lambat dalam membuat laporan total penjualan harian untuk setiap tenaga penjualan dan setiap kasir. Setelah dilakukan analisa oleh staf teknis, diketahui bahwa penyebab lambatnya kecepatan tersebut adalah karena total volume data tabel dokumen penjualan dan tabel detail dokumen penjualan telah mencapai 60GB. Diketahui struktur kedua tabel tersebut adalah sebagai berikut:
Tabel dokumen penjualan (nomor dokumen penjualan, waktu penjualan, nomor tenaga penjual, nomor kasir)
Detail dokumen penjualan (nomor dokumen penjualan, nomor produk, harga satuan, kuantitas)
Konsumsi CPU server database sangat tinggi saat melakukan perhitungan laporan ini. Untuk mengoptimalkan operasi ini, seorang insinyur menyarankan untuk menambahkan atribut "jumlah pembayaran total" ke tabel dokumen penjualan, dan nilainya secara otomatis dihitung oleh pemicu. Harap analisis pro dan kontra dari solusi ini dalam hal penggunaan ruang disk, dampak pada server database selama operasi penjualan, konsistensi data, dan dampak terhadap kecepatan penghitungan total penjualan. Harap menilai apakah solusi ini layak dan menganalisis alasannya dalam hal biaya waktu dan ruang serta efek pengoptimalan.

Pemicu adalah prosedur tersimpan khusus. Kekhususannya adalah tidak perlu dipanggil langsung oleh pengguna, tetapi dipicu secara otomatis ketika operasi UPDATE, INSERT dan DELETE dilakukan pada data dalam tabel. Pemicu biasanya digunakan untuk memastikan aturan bisnis dan kelengkapan data. Keuntungan utama mereka adalah pengguna dapat menggunakan metode pemrograman untuk mengimplementasikan logika pemrosesan dan aturan bisnis yang kompleks, yang meningkatkan fungsi batasan integritas data. Pemicu biasanya digunakan dalam situasi berikut: untuk menyelesaikan batasan data yang lebih kompleks daripada batasan CHECK, untuk memelihara data yang didenormalisasi untuk memastikan kinerja database, dan untuk menerapkan aturan bisnis yang kompleks. Pemicu juga dapat mengevaluasi status tabel sebelum dan sesudah modifikasi data, dan mengambil tindakan pencegahan berdasarkan perbedaannya. Dalam rencana ①, hanya desain pemicu yang memerlukan koordinasi oleh personel terkait. Jika jumlah datanya besar, hal ini dapat mempengaruhi pengoperasian sistem personalia, namun secara keseluruhan lebih baik dari rencana ②.

[Kang menjelaskan]
Opsi 1, ketika jumlah datanya besar, akan mempengaruhi pengoperasian tabel terkait dalam sistem personalia, namun akan berdampak kecil pada sistem manajemen perpustakaan. Efisiensi sistem secara keseluruhan akan ditingkatkan, dan kompleksitas koordinasi pengembang terkait akan menjadi kecil.
Opsi 2 akan mempengaruhi pengoperasian sistem personalia dan sistem manajemen perpustakaan. Basis data baru akan meningkatkan overhead sistem dan mempengaruhi efisiensi pengoperasian sistem secara keseluruhan. Kompleksitas koordinasi personel terkait lebih rendah dibandingkan Opsi 1.
Singkatnya, opsi 1 lebih baik daripada opsi 2. Karena opsi 2 hanya memerlukan kompleksitas koordinasi pengembang yang lebih rendah dibandingkan opsi 1.

alasan:
Teknologi pencadangan panas mesin ganda adalah solusi aplikasi yang sangat toleran terhadap kesalahan yang menggabungkan perangkat lunak dan perangkat keras. Solusinya terdiri dari dua sistem server, kabinet array disk bersama eksternal dan perangkat lunak pencadangan panas mesin ganda yang sesuai.
Dalam solusi toleransi kesalahan ini, sistem operasi dan aplikasi diinstal pada disk sistem lokal dari dua server, dan data seluruh sistem jaringan dikelola secara terpusat dan dicadangkan melalui susunan disk.
Manajemen data terpusat menggunakan sistem pencadangan panas mesin ganda untuk secara langsung membaca dan menyimpan data semua situs dari perangkat penyimpanan pusat dan mengelolanya oleh para profesional, yang sangat melindungi keamanan dan kerahasiaan data. Data pengguna disimpan dalam array disk bersama eksternal. Ketika server gagal, server cadangan mengambil inisiatif untuk mengganti host untuk memastikan layanan jaringan tidak terganggu.

(2) Mode operasi database arsitektur Active-Standby
Siaga panas mesin ganda secara khusus mengacu pada siaga panas (atau ketersediaan tinggi) berdasarkan dua server dalam sistem ketersediaan tinggi. Karena ketersediaan tinggi dua mesin umumnya digunakan di Tiongkok, maka disebut siaga panas dua mesin. Ketersediaan tinggi mesin ganda dibagi menjadi mode siaga aktif (mode Aktif-sSiaga) sesuai dengan metode peralihan selama bekerja. mode) dan mode dual-host (mode Aktif-Aktif). Modus aktif-siaga berarti bahwa satu server berada dalam status aktivasi layanan tertentu (yaitu, status Aktif), dan server lainnya berada dalam status siaga layanan (yaitu, status Siaga). Mode dual-host berarti dua layanan berbeda berada dalam status aktif dan siaga di dua server (yaitu, status Aktif-Siaga dan Siaga-Aktif).
Ada dua cara untuk menerapkan pencadangan panas mesin ganda. Salah satunya adalah dua server menggunakan perangkat penyimpanan bersama (biasanya array disk bersama atau jaringan area penyimpanan SAN) dan menginstal perangkat lunak mesin ganda untuk mencapai pencadangan panas mesin ganda, yang disebut metode bersama. Metode lainnya adalah tidak ada perangkat penyimpanan bersama, dan data disimpan di masing-masing server pada waktu yang sama, yang disebut metode perangkat lunak murni atau metode data tersinkronisasi perangkat lunak. Implementasi berdasarkan berbagi penyimpanan adalah solusi paling standar untuk hot standby mesin ganda. Dalam mode master-slave, kedua server menyediakan layanan eksternal dengan alamat IP virtual, dan permintaan layanan dikirim ke server aktif. Pada saat yang sama, kedua server mendeteksi status kerja server lain melalui jalur detak jantung. Setelah server utama gagal, server siaga akan membuat penilaian berdasarkan deteksi detak jantung, menyelesaikan peralihan dalam waktu singkat, mengambil alih semua sumber daya di host, dan menjadi server utama baru. Karena penggunaan perangkat penyimpanan bersama, kedua server sebenarnya menggunakan data yang sama, yang dikelola oleh perangkat lunak mesin ganda atau cluster.
