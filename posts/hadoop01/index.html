Bab 1 Ikhtisar Sistem File Terdistribusi (HDFS)

Kategori: Hadoop
Label:

Tanggal: 07-09-2022 12:18:31

HDFS adalah sistem file terdistribusi yang mendukung penyimpanan data besar-besaran dan merupakan pembawa penting untuk menyimpan file data besar. Bab ini terutama memperkenalkan arsitektur, mekanisme operasi, alur kerja, dan operasi dasar HDFS.

1. Ikhtisar HFDS

1. Rancang tujuan dan kekurangan HDFS

Tujuan desain:

  1. Mendukung penyimpanan file besar
  2. Mengadopsi model akses tulis-sekali-baca-banyak
  3. Dengan deteksi kesalahan dan fungsi pemulihan otomatis yang cepat
  4. Mengadopsi akses data streaming
  5. Mendukung komputasi seluler

tidak memadai:

  1. Tidak cocok untuk menangani akses data latensi rendah
  2. Ketidakmampuan untuk menyimpan file kecil dalam jumlah besar secara efisien
  3. Penulisan multi-pengguna dan modifikasi file secara sewenang-wenang tidak didukung

2. Arsitektur HDFS

HDFS adalah atuan/budakarsitektur, yaitu cluster HDFS yang terdiri dari NameNode dan beberapa DataNode komposisi. Mereka juga HDFSkomponen inti, di mana NameNode disebut "simpul metadata", yang lainnya disebut"simpul data”。

3. Blok data (blok)

Blok data disk adalah unit terkecil dari pembacaan dan penulisan disk, dan hanya data yang merupakan kelipatan integral dari ukuran blok yang dapat dibaca dan ditulis. Ukuran blok default untuk HDFS adalah 128MB。

Biasanya 512B, lalu kenapa settingannya besar sekali?Semakin besar blok datanya, semakin kecil pula overhead pengalamatannya.. Tentu saja, melakukannya secara berlebihan secara membabi buta tidak akan berhasil. Peta di MapReduce hanya memproses konten satu blok dalam satu waktu.Jika terlalu besar, keseluruhan volume tugas akan berkurang., sehingga mempengaruhi dataparalelKecepatan pemrosesan.

keuntungan:
File data dengan ukuran berapa pun dapat disimpan, asalkan diiris secara fisik.
Manajemen sistem penyimpanan yang disederhanakan: informasi metadata dan informasi data file dikelola secara terpisah.
Membantu mencapai toleransi kesalahan dalam sistem file terdistribusi: jika satu node gagal, blok lain membaca salinannya.
Hal ini kondusif untuk mencapai keseragaman beban dan meningkatkan keandalan cluster: jika ruang yang tersisa dari node data lebih rendah dari titik kritis, node data dipindahkan ke node data lain; selain itu, salinan blok data disusun di rak yang berbeda. Bahkan jika rak hilang, salinannya dapat dibaca, yang sangat meningkatkan keandalan cluster. Pada saat yang sama, deteksi kegagalan Hadoop dan kemampuan pemulihan otomatis yang cepat dapat memulihkan jumlah replika blok ini ke tingkat normal pada waktu yang tepat.

4. NameNode dan SecondaryNameNode

1) Simpul metadata (NameNode)

Kelola namespace sistem file HDFS. Faktanya, ini adalah struktur direktori HDFS. Pengguna dapat menambah, menghapus, memodifikasi, mengganti nama, dll. File HDFS.

Informasi metadata disimpan:

  1. Nama file, nama direktori, dan hubungan hierarki
  2. Pemilik dan izin direktori file
  3. Terdiri dari blok data apa setiap file?
  4. Blok data ke informasi pemetaan node data

Informasi metadata disimpan ke dua file di disk lokal:fsimage (file gambar metadata)Dan suntingan (file log transaksi)。

Ketika node metadata berjalan normal, semua operasi pembaruan ditulis ke file edit. Jika ditulis langsung ke file gambar metadata, lama kelamaan sistem akan melambat. Artinya, file editan akan menjadi semakin besar, yang tidak akan berdampak signifikan pada sistem, namun proses restart node metadata akan menjadi semakin lambat.

2) SecondaryNameNode (node ​​metadata tambahan)

Selesaikan masalah file editan yang terlalu besar. Ini adalah penggabungan berkala file cermin metadata dan file log transaksi. Karena penggabungan memerlukan konsumsi memori, biasanya keduanyaTidak pada satu titikdi atas.

keuntungan:

  1. Ini meningkatkan kinerja cluster, menyimpan informasi metadata dari node metadata, dan meningkatkan keamanan dan keandalan metadata sampai batas tertentu.
3) Node Data

Sebuah node data memiliki beberapa blok data, dan setiap blok data menyimpan salinan di beberapa node data, namun sebuah node data hanya dapat memiliki satu salinan.

memengaruhi:

Biasanya, node data membaca blok data dari disk, tetapi jika suatu blok sering diakses, sistem menyimpannya di memori node data.

2. Kesalahan dan Pemulihan Data

Salah satu tujuan utama HDFS adalah "memiliki fungsi deteksi kesalahan dan pemulihan otomatis yang cepat", yang memerlukan jaminan keandalan penyimpanan data bahkan jika terjadi kesalahan.常见的出错情况包括block 损坏、NameNode 和 DataNode 的错误。

1. memblokir pemrosesan kerusakan

Kesalahan transmisi jaringanDanKegagalan perangkat keras mesindan faktor lain dapat menyebabkan kerusakan data.
Ketika klien membaca file, ia akan memverifikasi setiap blok baca. Jika terjadi kesalahan, ia akan membaca blok data pada node data lain dan melaporkan blok kesalahan tersebut ke node metadata, yang kemudian akan mereplikasinya lagi.
Selain itu, setiap node data akan membuka amemblokir proses pemindaian, untuk memverifikasi kebenaran blok secara berkala, dan setiap kesalahan akan dilaporkan ke node metadata untuk diproses.

2. Penanganan kesalahan NameNode dan DataNode

1) Penanganan kesalahan NameNode

Informasi metadata disimpan di NameNode, dan hanya salinan ini yang unik, sehingga keamanan ini harus dipastikan.

Ada tiga metode toleransi kesalahan:

  1. Informasi metadata disimpan ke disk lokal dan disinkronkan ke NFS, namun metadata akan hilang karena bandwidth jaringan dan alasan lainnya.
  2. Jalankan SecondaryNameNode. Namun, karena informasi metadata cadangan ini tertinggal dari NameNode, beberapa informasi data juga akan hilang.
  3. Aktifkan NameNodes yang aktif dan cadangan.

3. Mekanisme pengoperasian HDFS

1. Mekanisme penyalinan

memengaruhi:
Demi menjaga cinta dan kedamaian...ahem, aku kesini lagi, datang lagi! Untuk memastikan toleransi kesalahan dan ketersediaan cluster.

2. Mekanisme detak jantung

Setelah NameNode dimulai, ia akan menunggu "detak jantung" dari semua DataNode: DataNode secara aktif mengirimkan "detak jantung" ke NameNode pada interval tertentu (default tiga detik) dan secara aktif melaporkan informasi statusnya sendiri.然后 NameNode 通过心跳向 DataNode 下达命令。
Jika tidak diterima dalam waktu lama, dapat membuktikan bahwa DataNode sedang down, lalu memeriksa informasi salinan blok pada DataNode dan membackupnya ke DataNode lainnya.
Omong-omong, DataNode akan mengirimkan "detak jantung" ke node data primer primer dan sekunder.

3. Penempatan replika dan strategi penginderaan rak

1) Salin penempatan

Ada beberapa rak dalam satu cluster, beberapa node data di setiap rak, dan setiap node data menyimpan banyak salinan blok. Selain itu, metadata dari node metadata menyimpan ID rak yang dimiliki setiap node data. Jadi, bagaimana cara mengalokasikan salinan blok file ke node data di cluster?

Secara default, jumlah replika yang dikonfigurasi adalah 3, dua di antaranya ditempatkan pada node data berbeda di rak yang sama dan satu ditempatkan di rak lain.

Umumnya, 3 sudah cukup. Jika lebih besar dari atau sama dengan 3, salinan berikutnya dapat ditempatkan secara sewenang-wenang. Hindari saja memiliki terlalu banyak replika persetujuan dalam satu rak.

2) Penginderaan rak (sadar rak)

Karena lokasi replika mempengaruhi keandalan dan kinerja HDFS, HDFS mengadopsi strategi sadar rak untuk meningkatkan keandalan data dan meningkatkan pemanfaatan bandwidth jaringan.
Dengan cara ini, meskipun satu rak gagal, keandalan data tidak akan terpengaruh karena replika di rak lain masih tersedia. Selain itu, saat membaca data, aplikasi dapat dibaca di beberapa rak secara bersamaan, sehingga sangat meningkatkan kecepatan membaca data.

4. Mekanisme federasi

Informasi metadata setiap file perlu disimpan di memori NameNode, sehingga ada mekanisme federasi.

memengaruhi:
Metode ekspansi horizontal cluster memecahkan masalah kemacetan NameNode, yaitu,Tingkatkan jumlah node metadata。
Karena Hadoop mengelola penyimpanan dan pengelolaan data berukuran besar dan mengadopsi metode penyimpanan data redundan,Disk I/O adalah "hambatan utama" cluster.

Dalam mekanisme federasi, setiap NameNode mengelola sebagian kaliber penamaan sistem file (volume namespace). Setiap volume menyimpan metadata namespace dan kumpulan blok blok data file. Pada saat yang sama, setiap volume tidak bergantung satu sama lain dan tidak mempengaruhi atau berkomunikasi satu sama lain. Selain itu, semua DataNode di klaster harus didaftarkan pada setiap NameNode.

Namun, satu titik masalah kegagalan belum terselesaikan. Jika NameNode gagal, NameNode tetap tidak dapat diakses.

5. Mekanisme HA (layanan 7*24 jam tanpa gangguan).

Yang disebut HA berarti ketersediaan tinggi (layanan tanpa gangguan 7*24 jam). Hal terpenting untuk mencapai ketersediaan tinggi adalah menghilangkan satu titik kegagalan.
Metode: Izinkan dua NameNode utama dan cadangan dijalankan. Ketika node NameNode gagal, NameNode cadangan dapat dengan cepat diaktifkan untuk memastikan operasi normal cluster.

Node data siaga dan node data primer selalu disinkronkan (informasi metadata konsisten), dan keduanya berkomunikasi melalui daemon JournalNode.

6. Mode Aman

Setelah diaktifkan, Anda masuk ke mode aman. Dalam mode ini, node data master memeriksa integritas blok. Selain itu, ini adalah mode hanya-baca.
Dalam keadaan normal, NameNode akan keluar dari mode aman saat dimulai (dengan tambahan penundaan 30 detik). Namun, jika blok data yang hilang oleh DataNode melebihi nilai yang ditetapkan, cluster akan selalu berada dalam mode aman.

dua hal

  1. Tunggu detak jantung setiap node data untuk menentukan apakah sedang down, lalu NameNode membandingkan laporan blok yang dikirim oleh DataNode dengan metadatanya untuk menentukan apakah blok data normal.
  2. Memuat fsimage ke dalam memori, lalu menggabungkan fsimage dan pengeditan menjadi fsimage baru, dan membuat pengeditan baru. Setelah penggabungan selesai, hapus fsimage lama dan editannya, lalu ganti nama yang baru.

    7. Pengumpulan sampah

    Salinan blok yang tidak memiliki nilai guna dianggap sampah: jika sebuah file dihapus, salinan cadangan file tersebut tidak berguna.
    Penghapusan tidak akan langsung menghapusnya, melainkan akan memindahkannya ke Recycle Bin.

    4. Alur kerja HDFS

    Dalam kluster Hadoop,klienDansimpul metadatakomunikasi antarsimpul metadataDansimpul datakomunikasi antarsimpul dataKomunikasi antara satu sama lain didasarkan pada RPC (Panggilan Prosedur Jarak Jauh)Mekanis.

1. Mulai prosesnya

Selama proses startup HDFS, NameNode dan DataNode harus dimulai.
Saat memulai, pertama-tama akan masuk ke mode aman.
Ketika node data dimulai, proses DataBlockScanner akan dimulai untuk memindai blok, dan proses tersebut secara berkala akan mengirimkan "detak jantung" ke setiap NameNode.

2. Proses membaca

Saat Klien membaca, Klien mengakses NameNode terlebih dahulu untuk mengonfirmasi apakah Klien dapat membaca. Jika ya, Klien memperoleh informasi blok dan DataNode dari file tersebut, dan kemudian melakukan operasi baca HDFS untuk mendapatkan data. Setelah membaca data, aliran input file harus ditutup.

3. Tulis prosesnya

Ketika Klien menyiapkan data untuk ditulis ke file HDFS, Klien akan mengakses NameNode terlebih dahulu untuk mengonfirmasi izin penulisan dan apakah file tersebut ada (apakah ditimpa atau tidak). Ya, Klien membuat informasi metadata yang ditulis ke file di NameNode, mengembalikan informasi blok dan DataNode yang dapat menyimpan data, dan kemudian melakukan proses replikasi replika berdasarkan informasi yang dikembalikan. Setelah penulisan selesai, aliran keluaran file harus ditutup.

4. Hapus proses

1) Hapus file menggunakan perintah HDFS

Direktori recycle bin pengguna root adalah hdfs://hadoop0:9000/user/root/.Trash。
Saat pengguna menggunakan perintah HDFS untuk melakukan operasi penghapusan, sistem akan memindahkan file yang perlu dihapus ke "/Saat Ini" di Recycle Bin. Misalnya, jika pengguna root menghapus file "koinl01.txt", Anda bisa hdfs://hadoop0:9000/user/root/.Trash/Current/koinl01.txt muncul.
Tempat sampah juga memiliki jangka waktu.

2) Hapus file menggunakan Java API

Perlu ada penundaan sebelum benar-benar dapat dihapus.
